1. Qué debe controlar la licencia

La licencia debe validar:

dominio
clave licencia
estado licencia
fecha expiración

Ejemplo de licencia:

PCC-OTEC-2026-AB12-CD34
2. Servidor de licencias

Tendrás un pequeño API propio.

Por ejemplo:

https://licencias.pccurico.cl/api/validate

El plugin enviará:

license_key
domain
plugin_version

Respuesta del servidor:

{
"status":"valid",
"expires":"2027-01-01",
"domain":"cliente.cl"
}

Si la licencia es inválida:

{
"status":"invalid"
}

4. Código base del sistema de licencia

Archivo:

includes/license.php

Código:

function pcc_validate_license($key){

$domain = $_SERVER['SERVER_NAME'];

$response = wp_remote_post(
'https://licencias.pccurico.cl/api/validate',
array(
'body'=>array(
'license_key'=>$key,
'domain'=>$domain
)
)
);

if(is_wp_error($response)){
return false;
}

$data = json_decode(
wp_remote_retrieve_body($response),
true
);

if($data['status']=="valid"){
return true;
}

return false;

}

Cuando el admin presiona activar licencia:

En el plugin principal:

agregas algo así:

$license = get_option('pcc_license_status');

if($license!="valid"){

add_action('admin_notices',function(){

echo "<div class='notice notice-error'>
<p>PCC-WooOTEC requiere activación de licencia</p>
</div>";

}

Ahora viene la decisión arquitectónica más importante de todo el plugin.

Y si esto se hace mal, todo se vuelve frágil.

Debemos decidir cómo se comunicará el plugin con Moodle.

Hay tres métodos posibles:

1️⃣ Moodle WebService API
2️⃣ Token REST API
3️⃣ API personalizada Moodle

Uno de esos es 10 veces más estable para este proyecto.

Los otros dos funcionan… hasta que dejan de funcionar justo cuando tienes 300 alumnos comprando cursos. Y ahí empiezan los insultos tecnológicos.

Así que el siguiente paso lógico es definir la arquitectura de conexión WordPress ↔ Moodle antes de seguir escribiendo código como si la gravedad no existiera.
